# Final Reflection: ITMO 453 Final Production Deployment Project

The deployment process of the final project entailed design, protection, automation, monitoring, and maintenance of a Linux-based publicly available service deployed on actual cloud infrastructure. Contrary to prior coursework projects, which were carried out in a virtual machine on a local computer and could simply be shut down upon completion of the assignment, this deployment consisted of making a Linux server publicly available through a proper domain name, actual TLS certificates, and unauthorized access attempts being logged within the first hours after deployment. The system should be up and running all the time without further active management by the operator. Successfully deploying the system and resolving the critical configuration problem that made it inaccessible publicly revealed several lessons about operating such infrastructure, which were not covered in the previous coursework. This paper will analyze this experience starting with the topic of operational development, then discussing the issues of security, infrastructure pros and cons, future improvement options, and, finally, the practical value of the experience obtained.

## Operational Growth

Starting with the aspect of operational development, it should be noted that previously, the coursework assignments were focused on building a system rather than its maintenance. For example, in one lab, Ubuntu Server was installed in VirtualBox, SSH hardening was performed, a firewall was configured, and after capturing the necessary screenshots, the work on this machine was done. The labs on monitoring carried out using Docker Desktop helped familiarize the use of Prometheus, Grafana, Loki, and Uptime Kuma. However, the latter worked just as long as the computer was powered on. Continuous work like this demanded a more systematic approach to troubleshooting compared to my previous tasks.

In case an error "Insufficient Capacity" occurred in connection with the provisioning of Oracle Cloud, a further examination showed that another compute shape with a different capacity pool was available. Thus, choosing the other option helped me solve the problem during the next attempt. Another example of a similar kind was that of a GitHub Actions workflow that did not get executed after the code push. It was thought to mean a wrong configuration of the workflow. However, when I examined the repository structure myself, I saw that the workflow file was in the wrong place because it was one directory below where it was expected to be, which caused GitHub Actions not to see it.

Another problem related to the above-mentioned one, but it was much bigger and could not be solved in a usual way. There was a problem with the validation of the certificate, which could not be fixed through a usual configuration checkup. While my cloud provider's security list, its route table, and my server's firewall list were all configured correctly, there was still no possibility to access port 80 from outside of the network. After several hours of analysis of all the components one by one, the kernel firewall table was finally looked into instead of looking into general status commands. It turned out that there was a default rule created by the cloud provider's image that refused connections before applying any firewall configurations. Removing that rule fixed the problem.

## Security Lessons

In addition to this list of lessons, there were some important observations concerning security. Just a couple of hours after the server became publicly available, there were login attempts to SSH coming from many different countries, and Fail2Ban started issuing bans on some of these IP addresses. It provided a practical example of what an attack surface is rather than just a concept. In this way, the approach towards security hardening became not only a set of required actions, but also an operational need, and this point affected the whole further process of configuration. This change is clearly demonstrated by the architecture that was used. There is a security list of the cloud provider itself that allows only traffic through three ports to the host in question. The firewall of the host itself performs the same restriction once more. Access to SSH is restricted to key-based authentication, disabling root accounts, and limiting the number of attempts. Fail2Ban checks SSH and reverse proxy logs and blocks IPs that attempt to access the server multiple times.

Moreover, in this case, the implementation of the security measures went further to include alerting and monitoring capabilities beyond just network configurations. The alert rules created were not just implemented and presumed to work properly without any other validations. Memory pressure was intentionally induced on the running server with a stress testing tool to see how the alert goes through its states in Grafana. Then an email was received with information about the breach of the particular threshold. Hence, it became evident that all components of the alerting pipeline worked fine. Similar testing was done on the backup system with the removal of one of the live data volumes and subsequent successful restore from the backup file.

## Infrastructure Tradeoffs

Aside from those security and monitoring results, a few tradeoffs have been identified and accepted intentionally in this project. First of all, the tradeoff was made concerning the choice of the computing platform, since Oracle has a lot of options in terms of free-tier compute shapes with a lot of memory and CPU power, but it uses ARM processors instead of x86. The problem with such an approach is that the two images created to run a load test failed because they were not compiled for ARM processors and did not work when being emulated. But still, the platform was chosen because of the possibility to operate continuously for free.

The second tradeoff pertains to the deployment of a single virtual machine without redundancy and high availability support. In this scenario, the whole platform depends on a single host, and there is no backup server ready to take its place in case the primary server fails to operate. This issue was resolved with a proven recovery process instead of creating a redundant and available infrastructure. The said recovery process was not just described but implemented once in a controlled environment, allowing us to restore functioning production data. The value of a proven process over a just-written one was more important in terms of the project size and timeline than the additional difficulty of having two servers.

The third tradeoff involves the configuration management, as the server provisioning in this particular project is carried out via a shell script instead of using a declarative configuration management tool like Ansible. While in the first scenario, we have a set of commands that are supposed to be run just once during the initial setup process, the latter approach involves describing the desired target state of our system. Since Ansible took longer to implement, it was not chosen for the configuration of this project. However, this particular limitation is a point that needs improvement for the next version of the project.

## Future Improvements

The very same constraint shows a way forward for developing this project further. The implementation of the provisioning process using Ansible would mean the ability to declaratively define the server configuration and reapply it instead of running it only once as a script. Using Terraform for managing cloud infrastructure would mean the ability to recreate the whole environment based on the defined configuration instead of doing it manually through the cloud console. It is also possible to extend the alert management with a special tool for handling alerts along with it because one can group alerts, silence them, set up more complex notifications, etc., when the operations become too much for a single person to handle. Finally, it would be useful to automate the weekly transfer of the backup files from the server to cloud storage.

## Real World Applicability

Taken as a whole, these results show that this deployment is not far from what is done professionally during infrastructure operations. The hardening of the Linux host, containers running behind the reverse proxy, and infrastructure definition in the version-controlled repository are all practices that are common in junior site reliability and operations positions. Automated deployment from version control changes, continuous monitoring of metrics and logs, and validation of alerting and backup practices support this claim. All of these do not represent a simplified approach to some task designed for the sake of coursework. Some of the issues encountered here, like the unnoticed firewall rule or the incorrect automation config file placement, are common for real infrastructure operations.

Furthermore, this project has made use of the knowledge acquired in previous assignments. In past assignments, the main emphasis was placed on the development and implementation of applications and services in a clean environment, where there was no external audience, nor any kind of ongoing operations to deal with. However, in the current project, the focus has shifted to running a similar kind of system within a real-world public network environment. Consequently, the system implemented through this project can be used as tangible proof of one's skills, and not simply as information regarding coursework.

In conclusion, the current project changed the definition of what qualifies as a working system. It is not enough to only make sure that a system functions properly. In addition, it must be determined how fast a problem will be identified in the future and what kind of steps would be necessary to bring the system back into operation. Moreover, it must be ensured that the process of recovery and notification is tested under realistic conditions and not just assumed to work properly because of a good design.
